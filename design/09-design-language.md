# デザイン言語

CumulogのUIデザインの原則を定義する。

本ドキュメントは原則レベルを対象とし、デザイントークンの具体値（色コード、フォントサイズ、余白の数値）、UIライブラリの選定、アイコンセット、ロゴ等のブランド要素は実装側（`app/` 配下）で管理する。

## デザイン原則

1. **記録と振り返りが主役** — 装飾はコンテンツ（活動ログ）の可読性を妨げない
2. **温かみ・親しみやすさ** — 趣味活動の楽しさと積み重ねの実感（[提供価値](../docs/02-users-and-value.md)）を支えるトーンとする
3. **警告は親しみやすさより明瞭性** — 公開確認・削除確認・ネタバレ表示は、トーンを優先して曖昧にしない
4. **色だけに意味を持たせない** — 意味のある表示は色・アイコン・文言のセットで伝える（[非機能要件: アクセシビリティ](../docs/06-non-functional.md)）
5. **短時間入力を妨げない** — 作成フォームは視覚的な演出より入力効率を優先する

## トーン

* 柔らかいニュートラルを基調に、温色系のアクセントカラーを用いる
* 角丸・余白を広めにとった、圧迫感のない画面とする
* UI文言は丁寧で柔らかい日本語とする。ただし警告文（公開確認・削除確認）は簡潔・直接的に書き、婉曲表現でリスクをぼかさない

## カラーの原則

* 色は役割ベースのセマンティックトークン（背景、テキスト、アクセント、警告、エラー、成功、ネタバレ等）として定義し、画面実装に生の色値を直書きしない
* 警告・エラーは一般的な慣習（警告=黄系、エラー・破壊的操作=赤系）に従い、温色系アクセントと混同しない色分離を保つ
* テキストと背景のコントラスト比はWCAG 2.1 AA（本文4.5:1以上、大きな文字3:1以上）を満たす
* ダークモードはMVPでは対象外とする。ただしセマンティックトークンで設計することで、将来のダークテーマ追加をトークン差し替えで実現できる構造にする

## 状態表現の原則

意味のある状態は、すべて「色＋アイコン＋文言」のセットで表現する。

| 状態 | 表現の原則 |
| --- | --- |
| 公開データの明示 | 専用のアイコンと文言を定め、S1・公開前確認・詳細画面で常に同一の表現を使う |
| 削除の警告 | エラー系（赤系）の色とアイコン。確定ボタンは取り消しと視覚的に明確に区別する |
| ネタバレラベル | 「軽微」「重大」の2段階をラベルとして表示し、重大は軽微より強い表現とする。「なし」はラベルを表示しない |
| 入力エラー | 該当項目の近くに、エラー系の色＋アイコン＋理由の文言で表示する |
| 取得・保存の失敗 | エラー表示と再試行操作をセットにする |
| 空状態 | 失敗と混同しない中立的な表現とし、最初の活動ログ作成への導線を添える |
| 成功 | 控えめに伝える。保存成功はダイアログではなく画面遷移とトースト等の軽い通知で示す |

## タイポグラフィ・レイアウトの原則

* 日本語はOS標準のシステムフォントスタックを用いる。WebフォントはMVPでは配信しない（表示速度と運用負荷のため）
* フォントサイズは補足・本文・見出しの3段階程度の小さなスケールで運用する
* スマートフォンでは単一カラムを基本とする（[画面設計](04-screens.md)）
* タップターゲットは44px四方以上を確保する
* 余白はひとつの基本単位の倍数で統一する

## コンポーネント方針

* UIコンポーネントは既存の軽量ライブラリを採用し、セマンティックトークンの適用で見た目を統一する
* ダイアログ・フォーム・トースト等のフォーカス管理・キーボード操作などのアクセシビリティ実装はライブラリに委ね、自前実装を避ける
* ライブラリの選定は実装側（`app/`）で行う
