# データフロー設計

## API設計の方針

* Cumulog独自のAPIは設けない
* すべての読み書きは、ログインユーザーのPDSに対する `com.atproto.repo.*` XRPCで行う

| 操作 | XRPC | 備考 |
| --- | --- | --- |
| 作成 | `com.atproto.repo.createRecord` | collection: `jp.mp0.cumulog.log`、rkeyはTID自動採番 |
| 一覧取得 | `com.atproto.repo.listRecords` | cursorで全件取得 |
| 詳細取得 | `com.atproto.repo.getRecord` | 一覧からの遷移では取得済みデータを再利用してよい |
| 編集 | `com.atproto.repo.putRecord` | `swapRecord` に取得時のCIDを渡す |
| 削除 | `com.atproto.repo.deleteRecord` | `swapRecord` に取得時のCIDを渡す |

## 一覧取得とソート

* `listRecords` はrkey順でしか返らないため、cursorを辿って自分の全活動ログを取得し、クライアント側でソートする
* ソートキーは活動日（`activityDate`）の降順、同一活動日内は作成日時（`createdAt`）の降順とする
* MVPの想定規模（個人利用、数十〜数百件）では全件取得・クライアントソートで成立する。件数増加時の段階取得・仮想化は実利用で問題が出た時点で導入する
* 取得結果はメモリ内のアプリ状態として保持し、画面間で再利用する。永続キャッシュは持たない（[全体構成](01-architecture.md)）
* 作成・編集・削除の成功後は、アプリ状態を更新して一覧へ反映する

## タグ絞り込み・検索方式

* タグ絞り込みは、取得済みの一覧に対するクライアント側フィルタとして実装する
* 判定はタグ文字列の完全一致（保存時にトリム済み）とする
* 検索インデックス・全文検索・サーバー側検索はMVPでは持たない
* 活動種別・対象名・年月による絞り込み（Should）を実装する場合も、同じクライアント側フィルタの追加として実現できる構成とする

## 編集・削除時の整合性

* 編集・削除は、レコード取得時のCIDを `putRecord` / `deleteRecord` の `swapRecord` に渡すCAS（compare-and-swap）方式とする
* 編集がCID不一致（他クライアント等による先行変更）で失敗した場合は、編集が成功したように表示せず、対象ログが変更されている旨を表示して再読み込みを促す。このとき入力内容は画面上に保持する
* 削除がCID不一致で失敗した場合は、削除確認で表示した内容と現在のレコードが異なるため削除を実行しなかった旨を表示し、再読み込み後に改めて削除確認をやり直せるようにする
* 削除は成功レスポンスを確認してから一覧・詳細の表示から取り除く
* 楽観的UI更新（成功前に画面へ反映する方式）は採用しない。保存・編集・削除は成功確認後に反映する（要件の「失敗時に成功したように表示しない」を構造的に満たすため）

## 他クライアント由来レコードの扱い

* 一覧取得時に各レコードを[Lexicon・データ設計](02-lexicon.md)のスキーマで検証する
* 必須項目を満たさないレコードは「読み込めない活動ログ」として一覧に表示し、削除のみ可能とする
* 旧形式・別形式のデータを暗黙に解釈して表示し続けるフォールバックは実装しない（リポジトリ方針に従う）。スキーマを変更する場合は、変更時に移行方法をあわせて設計する

## Bluesky共有（Should）

* 共有は活動ログ保存の成功後に、活動ログ詳細画面からの別操作として `app.bsky.feed.post` レコードを作成する。保存フローには組み込まない
* 既定文面は、タイトル・活動日・公開共有ページのURLから生成する。外部URLは公開共有ページのURLと併記すると分かりにくいため含めず、メモとタグはネタバレを含みうるため含めない。具体的な文言は実装判断に委ねる
* 投稿文面はユーザーが投稿前に確認・編集できる。上限はBluesky投稿の制約（300書記素・3000バイト）に従い、書記素数は `Intl.Segmenter` で数える
* 投稿時、確定した文面中の http / https URL を検出し、UTF-8バイトオフセットのlink facet（`app.bsky.richtext.facet`）を付与してリンクとして機能させる
* レコードは `$type`・`text`・`facets`（URLがある場合）・`langs: ["ja"]`・`createdAt` を持つ
* 共有の成否は活動ログレコードに保存しない（活動ログから投稿への参照は持たない）
* 共有の失敗は活動ログ保存の成否と区別して表示する（[エラー処理設計](06-error-handling.md)）

## 公開共有ページの取得（Should）

* 公開共有ページ（[画面設計 S8](04-screens.md)）は、URL中のDIDとrkeyから対象レコードを未認証で取得する
* DIDからDIDドキュメントを解決してPDSのサービスエンドポイントを取得し、`com.atproto.repo.getRecord` を認証なしで呼び出す（公開レコードの読み取りに認可は不要）
* 取得したレコードは活動ログと同じスキーマ検証を行い、検証を満たさない場合は表示できないログとして扱う
* 閲覧では対応PDS判定（Bluesky公式PDS限定）は行わない。判定はログイン時の書き込み対象の制限であり、公開データの閲覧には適用しない
* 所有者のhandleはDIDドキュメントから取得し、解決できない場合はDIDを表示する
* 書き込みは行わないため、タイムアウト・失敗は読み取りエラーとして扱う
