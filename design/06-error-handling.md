# エラー処理設計

## 方針

* 失敗を成功のように見せない。保存・編集・削除は成功確認後にのみ画面へ反映する（[データフロー設計](05-data-flow.md)）
* エラーは「何が起きたか」「ユーザーが次に何をできるか」を日本語で表示する。XRPCのエラーコードや例外メッセージをそのまま表示しない
* 失敗した操作の入力内容は、画面上に可能な限り保持し、再試行できるようにする

## エラー分類と表示

| 分類 | 検出契機 | 表示・挙動 |
| --- | --- | --- |
| 入力エラー | 保存前のクライアント検証 | 該当項目の近くに理由を表示する。公開前確認へ進めない |
| 認証失効 | トークンリフレッシュ失敗、XRPCの401 | 認証が切れた旨を表示し、再ログインへ誘導する。編集中の入力は画面上に保持する |
| 権限不足 | XRPCの403等 | 権限が不足している旨と、再ログインで解決する可能性を表示する |
| 保存・編集・削除の失敗 | createRecord / putRecord / deleteRecord のエラー、ネットワークエラー | 失敗した操作を明示し、再試行ボタンを表示する。入力内容・対象ログの表示を維持する |
| 編集・削除の競合 | putRecord / deleteRecord の swapRecord 不一致 | 対象ログが別の場所で変更されている旨を表示し、再読み込みを促す。編集中の入力内容は保持し、削除は再読み込み後に確認からやり直す |
| 一覧取得失敗 | listRecords のエラー | 「取得に失敗した」表示と再試行ボタンを表示する。0件の空状態表示（「まだ活動ログがない」）とは文言・見た目を区別する |
| 詳細取得失敗 | getRecord のエラー | 「取得に失敗した」表示と再試行を表示する。レコードが存在しない場合（削除済み等）は「見つからない」表示とし、取得失敗と区別する |
| 対応外PDS | ログイン時の判定 | MVPの対応対象外である理由を表示する（[認証・連携設計](03-auth.md)） |

## 再試行

* 再試行はユーザー操作によるものを基本とし、自動リトライは行わない（二重作成の防止を優先する）
* 作成の再試行で二重登録が起きた場合に備え、失敗時は「保存されたか不明」なケース（タイムアウト等）とそれ以外を可能な範囲で区別し、不明なケースでは一覧の再読み込みを促す文言を添える

## 外部サービス障害時

* PDS側の障害・仕様変更で操作が継続できない場合も、上記の分類（取得失敗・保存失敗等）として表示する
* 障害の原因究明に必要な情報（エラーコード等）は、表示メッセージの詳細としてユーザーが展開できる形で添えてよい
